## 一、选题背景与实验目的（2页）

### 1.1 选题背景

**操作系统课程的重要性**
- 计算机专业核心基础课程
- 涵盖进程管理、内存管理、CPU调度等关键概念
- 是理解计算机系统工作机制的关键环节

**传统教学的痛点**

| 问题 | 具体表现 |
|------|----------|
| 进程状态转换不直观 | 学生难以理解五状态模型的动态变化 |
| 调度算法执行流程抽象 | 时间片轮转、优先级等概念难以追踪 |
| 内存分配过程复杂 | 碎片产生机制不易观察 |
| 同步机制理解困难 | P/V 操作、死锁等逻辑抽象 |

**可视化教学的价值**
- 将抽象概念具象化，降低理解门槛
- 实时交互式演示，增强学习体验
- 支持算法对比，加深理解

### 1.2 实验目的

1. **理论理解**：通过可视化方式深入理解操作系统核心原理
2. **算法掌握**：掌握进程调度、内存管理等经典算法的实现
3. **工程实践**：提升 GUI 开发与软件工程实践能力
4. **综合应用**：将多线程编程、事件驱动等技术综合运用

## 二、系统总体架构与模块划分（2页）

### 2.1 技术选型

| 技术 | 选型 | 选型理由 |
|------|------|----------|
| 开发语言 | Python 3.8+ | 开发效率高，生态丰富，跨平台 |

### 2.2 系统架构（MVC 分层）

```
┌─────────────────────────────────────────────────────┐
│                    UI 层 (Controller)                │
│  main_window.py + 各功能模块界面 (*_module.py)        │
├─────────────────────────────────────────────────────┤
│                可视化组件层 (View)                    │
│  state_machine / gantt_chart / memory_view / ...    │
├─────────────────────────────────────────────────────┤
│                  核心逻辑层 (Model)                   │
│  process / scheduler / memory / synchronization     │
└─────────────────────────────────────────────────────┘
```

### 2.3 模块划分

| 模块 | 核心功能 | 对应文件 |
|------|----------|----------|
| 进程管理 | 进程生命周期、状态转换、线程管理 | `process.py`, `process_module.py` |

| 任务管理器 | 系统实时监控 | `system_monitor.py`, `task_manager_module.py` |

## 三、各模块的设计原理与实现方法（8-10页）

### 3.1 进程管理模块（1-2页）

**设计原理**
- 采用五状态模型：新建 → 就绪 → 运行 → 阻塞 → 终止
- 使用状态机模式实现状态转换逻辑
- 支持进程与线程两级管理

**状态转换图**
```
    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
    │ 创建 │───▶│ 就绪 │◀──▶│ 运行 │───▶│ 终止 │
    └──────┘    └──────┘    └──────┘    └──────┘
                    ▲           │
                    └───────────┘
                         阻塞
```

**实现要点**
- `Process` 类封装进程属性（PID、状态、优先级等）
- `ProcessManager` 管理进程队列与状态转换
- Qt 信号槽实现 UI 与逻辑解耦

**可视化展示**
- 状态机图：圆形节点表示状态，箭头表示转换
- 就绪/阻塞队列动态更新

### 3.2 进程通信模块（1页）

**设计原理**
- 经典生产者-消费者问题
- 有界缓冲区实现共享内存通信
- 信号量实现同步互斥

**核心算法**
```
生产者：                    消费者：
  P(empty)                    P(full)
  P(mutex)                    P(mutex)
  放入产品                    取出产品
  V(mutex)                    V(mutex)
  V(full)                     V(empty)
```

**可视化展示**
- 缓冲区格子动态填充/消费动画
- 生产者/消费者状态指示
- 实时事件日志

### 3.3 进程同步模块（1-2页）

```
偶数号哲学家：先拿左叉，后拿右叉
奇数号哲学家：先拿右叉，后拿左叉
→ 打破循环等待条件
```

**可视化展示**

### 3.4 CPU 调度模块（1-2页）

**设计原理**
- 实现4种经典调度算法
- 甘特图展示调度过程
- 性能指标对比分析

**算法对比**

| 算法 | 原理 | 特点 |
|------|------|------|
| FCFS | 按到达顺序调度 | 简单，但可能导致"护航效应" |
| RR | 时间片轮转 | 公平，响应时间短 |
| SJF | 选择剩余时间最短的进程 | 平均等待时间最优 |
| Priority | 按优先级调度 | 灵活，但可能导致饥饿 |

**可视化展示**
- 甘特图动态绘制调度过程
- 进程颜色编码
- 性能指标面板：平均等待时间、平均周转时间

【截图位置：CPU调度甘特图界面】

### 3.5 内存管理模块（2页）

#### 3.5.1 动态内存分配

| 算法 | 原理 | 优缺点 |
|------|------|--------|
| 首次适应 | 从头找第一个满足的空闲块 | 简单，但前端碎片多 |
| 最佳适应 | 找最小满足的空闲块 | 碎片小，但查找慢 |
| 最坏适应 | 找最大的空闲块 | 减少小碎片，但大块消耗快 |
| 循环首次适应 | 从上次位置继续查找 | 分布均匀 |

【截图位置：内存分配界面】

#### 3.5.2 页面置换

| 算法 | 原理 | 特点 |
|------|------|------|
| FIFO | 淘汰最先进入的页面 | 简单，可能Belady异常 |
| LRU | 淘汰最近最少使用的页面 | 性能好，实现复杂 |
| OPT | 淘汰未来最长时间不用的页面 | 最优，但不可实现 |
| CLOCK | 时钟指针扫描 | 近似LRU，开销小 |

## 六、问题与改进方向（1-2页）

### 6.1 遇到的问题与解决方案

| 问题 | 原因分析 | 解决方案 |
|------|----------|----------|
| 多线程UI更新崩溃 | 非主线程直接操作UI | 使用Qt信号槽机制跨线程通信 |
| 任务管理器卡顿 | 主线程获取进程列表耗时 | 后台线程采集 + 定时器刷新 |
| 动画不流畅 | 重绘频率控制不当 | 使用QTimer控制帧率 |
| 死锁检测不准确 | 资源图分析逻辑错误 | 实现完整的环检测算法 |

- 界面美观度有待提升
- 部分算法参数不支持自定义

### 6.3 改进方向

**功能扩展**

- 支持自定义参数导入/导出
